How to test a sort function — exhaustively.

When the values a sort can see come from a finite, limited domain, you don't have to sample inputs and hope. You can recursively generate every input up to a bound and prove the sort correct over that whole space. This guide shows the generators, the correctness oracle, the metrics that tell you how far you can push, and live visualizations — with copyable code in six languages.

01Why sorts are hard to test well

A sort is one line to describe and surprisingly easy to test badly. The usual mistake is checking a handful of hand-picked inputs against hand-written expected outputs.

That style — assert sort([3,1,2]) == [1,2,3] — documents intent, but its coverage is exactly whatever cases you happened to imagine. It systematically misses the two things that actually break sorts:

  • Lost, duplicated, or invented elements. A sort that quietly drops a duplicate key still returns something ordered. If your assertion only eyeballs "looks sorted," it passes.
  • Order-dependent bugs. Off-by-one boundaries, bad pivots, and unstable merges surface only on particular arrangements — reverse order, a single out-of-place element, all-equal runs.

The fix has two moves. First, stop comparing against fixed outputs and instead check the properties every correct sort must satisfy (the oracle, §02). Second — the heart of this guide — when the element values are drawn from a small finite set, enumerate every possible input by recursion instead of sampling. Total coverage over a bounded space is a proof, not a guess.

★

The core idea. Finite value domain + bounded length ⇒ a finite set of possible inputs ⇒ generate and test all of them. Recursion is what makes "all of them" tractable to produce.

02The correctness oracle

Before you can test every input, you need a check that decides — for any input — whether an output is correct, without knowing the "answer" in advance.

A function sort is correct on input xs when its result ys satisfies both of these, and either one alone is a trap:

  1. Ordered. ys[i] ≤ ys[i+1] for every adjacent pair (under your comparator).
  2. Permutation of the input. ys contains exactly the same elements as xs, with the same multiplicities — a multiset equality.
!

Neither property is sufficient alone. Returning [0,1,2,…,len-1] is always ordered but almost never a permutation. Returning the input untouched is always a permutation but rarely ordered. Only the conjunction pins down "sorted."

The multiset check is the one people skip, and it's exactly the one that catches dropped or duplicated elements. Compare element counts (a histogram / Counter / multiset), not sorted-equality — using a second sort to verify your sort is circular.

Two more properties are cheap and worth asserting when they apply: idempotency (sort(sort(xs)) == sort(xs)) and stability (equal keys keep their input order — testable with (key, tag) pairs, see §11).

04Finite value domains: generate every sequence

If each element comes from an alphabet of k symbols, then the length-n inputs are the k-ary Cartesian product — exactly kⁿ sequences. Recursion enumerates them all.

"Finite / limited values" describes a huge share of real sorts: booleans (k=2), an enum or status code (k=3–8), small integers, priority buckets, a handful of key classes, bytes (k=256). Pick a small representative alphabet — the classic choices are {0,1} and {0,1,2} — and enumerate every sequence up to some length L. That covers 1 + k + k² + … + k^L inputs with zero gaps.

The generator is a backtracking walk: choose a symbol for position 0, recurse to fill the rest, then undo the choice and try the next symbol. The recursion tree below is the algorithm — each root-to-leaf path spells one sequence, and a depth-first walk yields them in generation order.

Why sequences and not just distinct orderings? Real bugs hide in repeated values (equal keys) and in order together. Sequences over a small alphabet exercise both at once — duplicates and arrangement — which is why they are the workhorse of finite-domain sort testing.

05All permutations, up to a maximum size

When you care about ordering of distinct keys, enumerate all n! permutations of a fixed n-element set — for n up to a feasible maximum.

Permutations stress the comparator and the movement logic: reverse-sorted input, rotations, a single element carried across the array. They're the natural exhaustive test for the arrangement dimension. The generator is again backtracking — pick an unused element for each position via a used[] mask.

★

Permutations never test duplicate handling. Every permutation of distinct keys has each key exactly once, so a sort that mishandles equal elements can pass all n! permutations and still be broken. To catch that class of bug you need repeated values — i.e. finite-alphabet sequences (§04). The live tester in §09 lets you watch this happen.

06Combinations with repetition (multisets)

A third enumeration sits between sequences and permutations: the distinct multisets of size n over k symbols — "how many of each value," ignoring arrangement. There are C(n+k−1, k−1) of them.

Generate them as non-decreasing sequences: recurse with a start index that never moves backward, so 0,0,1 is produced but its rearrangements 0,1,0 and 1,0,0 are not. This is combinations-with-repetition, and it's far smaller than kⁿ.

Multisets are useful in two ways. As canonical shapes: enumerate each multiset, then feed it plus a few of its permutations — a cheap way to cover "value distribution" without the full kⁿ. And as an exhaustive test in their own right — but only for sorts whose behavior is genuinely independent of input order.

!

Don't assume order-independence. Testing one representative per multiset silently assumes the sort treats all arrangements of the same values identically. Most real implementations don't — that's exactly where order-dependent bugs live. For a rigorous finite-domain proof, test all sequences (kⁿ). Reach for multisets when you want a smaller, complementary pass.

The four enumerations grow at very different rates: 2ⁿ (binary sequences), kⁿ (k-ary sequences), n! (permutations), and C(n+k−1,k−1) (multisets). 08The 0–1 principle: reduce everything to binary

One of the most useful results in sorting theory turns a huge test space into a tiny one — but only for a specific class of sorts, so read the scope carefully.

Knuth's 0–1 principle. If a sorting network — a fixed, data-independent sequence of compare-exchange operations — correctly sorts every sequence of 0s and 1s, then it correctly sorts sequences of arbitrary values. So to verify an n-wire network you test only 2ⁿ binary inputs instead of n! permutations or kⁿ sequences.

The intuition: a network is oblivious — it performs the same comparisons regardless of the data. For any monotonic threshold function that maps values to 0/1, the network commutes with that mapping. If some input weren't sorted, thresholding just below the first out-of-order value produces a 0/1 input the network also fails — so testing all 0/1 inputs is enough.

!

Scope. The 0–1 principle applies to sorting networks and any data-independent comparator sequence. It does not directly apply to data-dependent algorithms (quicksort, mergesort, timsort), whose comparison pattern changes with the values — a binary-only pass can miss bugs that only trigger on distinct keys. That said, binary inputs are so cheap that running them against any sort is still a high-value first pass; just don't treat it as a proof for data-dependent code.

09The exhaustive harness — and a live counterexample finder

Combine the oracle (§02) with a generator (§04–06) and you have a harness that proves a sort correct over a bounded space — and, when it isn't, hands you the smallest broken input.

The loop is simple: for each length from 0 up to L, for each generated input, assert ordered and permutation; on the first failure, report the input. Iterating by increasing length means the first counterexample you hit is a minimal-length one (§10).

10Minimal counterexamples (shrinking)

A failing input of length 3 teaches more than a failing input of length 300. Exhaustive-by-increasing-size gives minimality for free; randomized testing needs to shrink.

Because the harness enumerates length 0, then 1, then 2…, the first failure it reports already has minimal length. Among inputs of that length you can further prefer the lexicographically smallest, which the natural generation order gives you. That's a clean, human-readable witness — often something like [1, 0, 0].

Randomized testing (which finds a big, messy failing input) needs an explicit shrink: repeatedly simplify the counterexample — drop an element, pull a value toward the minimum — keeping any change that still fails, until no single step helps. The result is a local minimum that's usually tiny. This is what Hypothesis/QuickCheck do automatically.

i

Shrinking pairs naturally with the finite-domain mindset: values are already pulled toward a small alphabet, so the only axis left to minimize is length. A shrunk counterexample from a random run is frequently a case your exhaustive pass would have caught — confirmation that the two strategies reinforce each other.

11Edge cases & extra properties

Exhaustive small-input testing subsumes most hand-written edge cases — but a few structural inputs and a few extra properties still deserve explicit checks.

Always include these inputs

Enumerating lengths from 0 upward already covers them, but assert them by name too, as documentation and as fast smoke tests: the empty list, a single element, two elements, all-equal values, already-sorted, and reverse-sorted. Also test large runs of duplicates and, for numeric sorts, boundary values (min/max, and for floats the tricky ones — negative zero, and a decision about NaN).

Stability

A sort is stable if equal keys keep their input order. Test it by sorting (key, tag) pairs on key only, where tag is the original index; the output tags within each equal-key group must be increasing. Finite alphabets make this exhaustive too: with keys from {0,1,2} and distinct tags, enumerate all sequences and verify group order.

Comparator contract

If your sort takes a custom comparator, a subtly wrong one (not antisymmetric or not transitive) can corrupt results or throw — real libraries detect this, e.g. timsort's "comparison method violates its general contract." When the key domain is finite, you can exhaustively verify the comparator itself: check antisymmetry and transitivity across all triples of values before trusting it.

12A practical recipe

Put it together into a suite that is deterministic where it can be, probabilistic where it must be, and minimal in what it asks you to maintain.

  1. Write the oracle once. is_sorted + multiset equality, reused everywhere. Add idempotency and stability where relevant.
  2. Exhaust binary and ternary sequences up to the largest length your budget allows (§07) — typically n ≈ 20+ for binary, ≈ 15 for ternary. This is your deterministic core and catches duplicate-handling bugs.
  3. Exhaust all permutations up to n ≈ 8–10 for the ordering dimension.
  4. Randomize for scale — large lengths and wide value ranges — with automatic shrinking (§10) to keep failures readable.
  5. Pin named edge cases (§11) and any historical regressions as fixed examples.
  6. If the sort is a network / oblivious, use the 0–1 principle (§08): exhaust 2ⁿ binary inputs and you're done for that length.
